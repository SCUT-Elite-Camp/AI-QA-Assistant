## 2. 解析层

**代码位置：** `parsers/`

**入口：** `parsers/registry.py` → `parse_file(file_path)` → `Document`

### 2.1 架构设计

采用**策略模式 + 注册表**：添加新文件格式只需新建一个 Parser 类，无需修改任何已有代码。

```
parse_file(file_path)
    │
    ├── 根据扩展名查找 Parser
    │       ├── .pdf  → PDFParser
    │       └── .docx → DocxParser
    │
    └── 调用 parser.parse(file_path) → Document
```

### 2.2 PDF 解析 (`parsers/pdf_parser.py`)

**技术：** PyMuPDF (fitz)

**两遍处理流程：**

1. **第一遍 —— 提取结构信息**
   - 逐页扫描段落，记录每个段落的字体大小、是否加粗
   - 统计全文字号分布，计算中位数字号作为基准
   - 标记页眉页脚区域（上下 15%），后续跳过

2. **第二遍 —— 分类并生成结构化块**
   - **表格：** 调用 `page.find_tables()` 自动检测，转为 `ContentBlock(block_type=TABLE)`
   - **标题：** 加粗 + 字号大于中位数 1.2 倍 → `ContentBlock(block_type=HEADING, level=...)`
   - **列表：** 以列表符号开头 → `ContentBlock(block_type=LIST)`
   - **段落：** 其余文本 → `ContentBlock(block_type=PARAGRAPH)`

最后所有块拼接成完整 Markdown 文本（`Document.content`）。

### 2.3 DOCX 解析 (`parsers/docx_parser.py`)

**技术：** python-docx

DOCX 天然就是结构化格式，解析更直接：

- **样式识别：** `Heading 1` ~ `Heading 9` 样式直接映射为标题层级
- **表格：** 通过 `doc.tables` 与段落位置交叉索引，保持阅读顺序
- **列表：** 检查 XML 中的 `numPr` 标签判断是否为列表项
- **正文提取：** `iter_inner_content()` 按段落/表格交替顺序提取

### 2.4 输出：Document 对象

```python
Document(
    doc_id="31dcd057...",            # 文件路径的 MD5
    title="Agent_RL_Papers",         # 文件名（去掉扩展名）
    content="# Agent RL...\n\n...",  # 全文 Markdown
    space="测试空间",                 # 来源文件夹
    address="/abs/path/to/file",     # 原始文件绝对路径
    last_updated="2026-06-22T...",   # 文件修改时间
    content_blocks=[                 # 结构化块列表
        ContentBlock(block_type=HEADING, level=1, text="一、推理增强 RL"),
        ContentBlock(block_type=PARAGRAPH, text="DeepSeek-R1 首次在..."),
        ContentBlock(block_type=TABLE, rows=[...], headers=[...]),
        ...
    ]
)
```

## 3. 分块层

**代码位置：** `pipeline/chunker.py`

**入口：** `chunk_from_blocks(blocks, doc_id, chunk_size=500, overlap=100)` → `list[Chunk]`

### 3.1 为什么需要分块

- **Embedding 模型有输入长度限制**（BGE-small-zh-v1.5 最大 512 token）
- **检索精度：** 小块粒度更精确，大块上下文更多。500 字是中文 RAG 的经验甜点
- **LLM 上下文窗口有限：** 不能把整个文档塞给 LLM，需要只送最相关的片段

### 3.2 分块算法

采用**块感知的滑动窗口**，不是简单按字符切：

```
┌──────────────────────────────────────────────┐
│  文档内容块序列                               │
│  [标题1] [段落A] [表格X] [段落B] [标题2] ...  │
└──────────────────────────────────────────────┘
                    │
    逐个累积块，当累计长度逼近 chunk_size 时：
                    │
    ├── 如果当前块是表格 → 作为一个独立 chunk（不切断表格）
    ├── 如果当前块是标题 → 回退一行，让标题成为下一个 chunk 的开头
    └── 否则 → 在此处截断
                    │
    截断后：
    ├── 保存当前 chunk
    └── overlap 部分（前一个 chunk 尾部 100 字）衔接到新 chunk 头部
```

### 3.3 Chunk ID 格式

```
{doc_id}_chunk_{index}
```

例如：`31dcd057215b9503e76bb686cd5f9d42_chunk_0`

这是**全局唯一**的标识，在 Milvus、BM25、JSON 三个存储中保持一致，后续检索结果通过 chunk_id 跨系统关联。

### 3.4 输出示例

```python
[
    Chunk(index=0, chunk_id="31dcd057..._chunk_0",
          text="Agent 强化学习前沿论文综述\n本文档系统整理了..."),
    Chunk(index=1, chunk_id="31dcd057..._chunk_1",
          text="...DeepSeek-R1 首次在大规模语言模型上..."),
    ...
]
```

## 4. 向量化层

**代码位置：** `pipeline/embedder.py`

**入口：** `embed_texts(texts)` → `list[list[float]]`

### 4.1 双模式设计

```python
if 设置了 OPENAI_API_KEY:
    → 调用 OpenAI 兼容接口
    → 使用 text-embedding-3-small
    → 输出 1536 维向量
else:
    → 加载本地模型 BAAI/bge-small-zh-v1.5
    → 纯离线运行
    → 输出 512 维向量
```

### 4.2 本地模型详情

| 属性 | 值 |
|------|-----|
| 模型名称 | BAAI/bge-small-zh-v1.5 |
| 大小 | ~95MB |
| 维度 | 512 |
| 最大输入 | 512 token |
| 下载方式 | 优先 HuggingFace，失败时自动回退 ModelScope |
| 加载方式 | 懒加载 + LRU 缓存，进程生命周期内只加载一次 |

### 4.3 向量化的意义

文本 → 向量后，语义相似的文本在向量空间中距离更近。比如：

```
"如何配置 Milvus？"    →  vec_a [0.12, -0.34, 0.56, ...]
"Milvus 的安装步骤"     →  vec_b [0.11, -0.32, 0.58, ...]  ← 与 vec_a 接近
"今天天气很好"          →  vec_c [-0.45, 0.78, -0.23, ...] ← 与 vec_a 很远
```

检索时，把用户问题向量化后，在 Milvus 中通过向量距离找到最相似的 chunk。

## 5. 存储层

分块和向量化完成后，数据**三路并行写入**：

### 5.1 Milvus 向量存储 (`storage/milvus_store.py`)

**用途：** 语义相似度检索（ANN 近似最近邻）

```
Collection: doc_chunks
Schema:
┌──────────────┬────────────────────┬──────────────────────┐
│ 字段         │ 类型               │ 说明                 │
├──────────────┼────────────────────┼──────────────────────┤
│ id           │ INT64 (auto PK)   │ 自增主键              │
│ embedding    │ FLOAT_VECTOR(512)  │ BGE 向量              │
│ chunk_id     │ VARCHAR(256)       │ 全局分块 ID           │
│ chunk_text   │ VARCHAR(65535)     │ 分块文本 (检索时返回)  │
│ doc_id       │ VARCHAR(128)       │ 所属文档 ID           │
│ chunk_index  │ INT32              │ 分块序号              │
│ source_url   │ VARCHAR(512)       │ 来源链接（预留）       │
└──────────────┴────────────────────┴──────────────────────┘

索引: HNSW，内积度量 (IP)，M=16, efConstruction=200
搜索参数: nprobe=10
```

**关键点：** `chunk_text` 字段直接存储在 Milvus 中，检索时无需再查 JSON，一次查询即可拿到向量分数 + 文本 + 元数据。

### 5.2 JSON 文档存储 (`storage/document_store.py`)

**用途：** 完整文档元数据，检索结果补全信息时使用

```
存储路径: data/documents/{doc_id}.json
键: doc_id（文件路径的 MD5）
```

**JSON 结构（与 Document 模型一致）：**

```json
{
  "doc_id": "31dcd057215b9503e76bb686cd5f9d42",
  "title": "Agent_RL_Papers",
  "content": "# Agent RL...\n\n...(全文 Markdown)...",
  "space": "测试空间",
  "address": "D:\\my_agent\\RAGent\\data\\raws\\测试空间\\Agent_RL_Papers.docx",
  "last_updated": "2026-06-22T09:04:00+00:00",
  "chunks": [
    {
      "index": 0,
      "text": "Agent 强化学习前沿论文综述\n本文档系统整理了...",
      "chunk_id": "31dcd057..._chunk_0"
    },
    ...
  ],
  "source_url": "",
  "content_blocks": [...]
}
```

**为什么额外存 JSON？**
- Milvus 存的是"行"，不存完整文档结构
- JSON 包含全文、标题、分块列表等元数据，用于在检索结果中展示"来源文档：XXX"
- 本地 JSON 不需要 Milvus 在线即可读取，容灾简单

### 5.3 写入顺序

```
对每个文档：
  1. embed_texts(chunk_texts) → vectors
  2. milvus.insert_chunks(vectors, ...) → Milvus
  3. save_document(doc_id, data)       → JSON

全部文档处理完后：
  4. BM25Index.build_from_documents() → BM25 pickle
```

## 6. 索引层

**代码位置：** `retrieval/bm25_index.py`

**入口：** `BM25Index` 类

### 6.1 BM25 简介

BM25 是经典的关键词检索算法，衡量的是**词语匹配度**：
- 查询词在文档中出现的频率越高 → 分数越高
- 查询词在所有文档中出现得越频繁（太常见） → 权重越低
- 文档越长 → 给一个惩罚系数

### 6.2 在本项目中的实现

```
data/documents/*.json
        │
        ▼
  load_document(doc_id)
        │
        ▼
  提取每个 chunk 的 text
        │
        ▼
  jieba 中文分词 (每个 chunk → 词语列表)
        │
        ▼
  BM25Okapi 构建倒排索引
        │
        ▼
  pickle 序列化 → data/bm25_index.pkl
```

### 6.3 使用方式

```python
from retrieval.bm25_index import BM25Index

bm25 = BM25Index.load_from_file("data/bm25_index.pkl")
results = bm25.search("如何配置 Milvus", top_k=5)
# → [{chunk_id, doc_id, chunk_index, text, score}, ...]
```

### 6.4 为什么需要 BM25 + 向量检索并存

| 检索方式 | 擅长 | 短板 |
|---------|------|------|
| 向量检索 (Milvus) | 语义相似："配置"≈"安装"≈"设置" | 精确匹配弱，专有名词可能漏 |
| 关键词检索 (BM25) | 精确匹配："GRPO"就是"GRPO" | 同义词、改写、跨语言 |

**混合检索 = 两者互补。** 这也是 `retrieval/search.py` 要做的事——RRF 融合两路结果。

## 7. 编排层

**代码位置：** `pipeline/process.py`

**入口：**
- 命令行：`python -m pipeline.process <folder_path>`
- Python：`from pipeline.process import process_folder`

### 7.1 完整执行流程

```
process_folder("data/raws/测试空间", chunk_size=500, overlap=100)
│
├── 1. 扫描文件夹
│      os.listdir → 过滤出支持的扩展名 (.pdf, .docx)
│      → 找到 N 个文件
│
├── 2. 初始化 Milvus 连接
│      MilvusStore(host="localhost", port="19530")
│
├── 3. 对每个文件执行:
│   │
│   ├── 3.1 解析: parse_file(file_path) → Document
│   │       输出: doc_id, title, content, content_blocks
│   │       日志: "✅ 解析完成，全文 XXXX 字符"
│   │
│   ├── 3.2 分块: chunk_from_blocks(blocks, doc_id) → list[Chunk]
│   │       Document.chunks = chunks
│   │       日志: "✅ 切片完成，共 N 个分块"
│   │
│   ├── 3.3 向量化: embed_texts([chunk.text for chunk in chunks]) → vectors
│   │       日志: "⏳ 正在向量化 N 个分块..."
│   │
│   ├── 3.4 写入 JSON: save_document(doc_id, document.dict())
│   │       日志: "✅ JSON 已保存"
│   │
│   └── 3.5 写入 Milvus: milvus.insert_chunks(vectors, ...)
│           日志: "✅ 向量已写入 Milvus"
│
├── 4. 构建 BM25 索引
│      bm25.build_from_documents()
│      bm25.save("data/bm25_index.pkl")
│      日志: "BM25 索引构建完成，共 N 个分块"
│
└── 5. 返回: list[Document]
```

### 7.2 doc_id 与幂等性

```python
doc_id = MD5(文件绝对路径)
```

这意味着：
- **同一文件重复处理 → 同一个 doc_id → 覆盖更新**
- **文件移动到不同目录 → 不同 doc_id → 视为新文档**
- **管道中断后重跑 → 已处理文件被覆盖，数据一致**

## 8. 关键设计决策

### 8.1 为什么分块和向量化在 pipeline 中而不是在检索时

**一次处理，多次查询。** 分块和向量化是最耗时的步骤（特别是 embedding 调用），放在离线管线中完成，在线检索时只需要做一次 query 向量化即可快速返回。

### 8.2 为什么 chunk_text 存了两份（Milvus + JSON）

- **Milvus 中的 chunk_text：** 用于检索结果直接返回，无需二次查询
- **JSON 中的 chunks：** 用于完整文档浏览、调试、重建索引
- 代价是存储冗余，但换来了检索性能（少一次 JSON 读 I/O）

### 8.3 为什么 BM25 要整个重建而不是增量更新

`rank_bm25.BM25Okapi` 不支持增量更新——因为它需要知道整个语料库的词频分布来计算 IDF。每次新增或删除文档后，必须全量重建。对于知识库场景（文件变更不频繁），这是可以接受的。

## 9. 常见问题

### Q: 跑管线需要联网吗？

如果没设 `OPENAI_API_KEY`，使用本地 BGE 模型。首次运行需要下载模型（~95MB），之后纯离线。

### Q: 向量维度变了怎么办（比如从 OpenAI 1536 换成 BGE 512）？

需要先删除旧的 Milvus Collection：

```python
from storage.milvus_store import MilvusStore
store = MilvusStore()
store.delete_collection("doc_chunks")
```

然后重跑管线。`init_collection()` 会根据首次插入的向量维度自动创建匹配的 schema。

### Q: 怎么增量添加新文档？

直接把新文件放到 `data/raws/` 下，再跑一次 `process_folder`。已处理的文件会被覆盖（幂等），新文件会被追加。但 BM25 索引会完全重建。

### Q: 管线中断了怎么办？

直接重跑即可。`doc_id` 基于文件路径的 MD5，已完成的步骤会被覆盖，不会产生脏数据。
